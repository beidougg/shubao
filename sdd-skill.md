# Spec-Driven Development (SDD) 编程 Skill

## 核心理念

> **先定义 WHAT，再让 AI 做 HOW。** 将规格说明（Specification）作为唯一真实来源（Single Source of Truth），代码作为其派生产物。

## 适用场景

- AI 辅助编程（Claude、GPT、Gemini 等）
- 中大型项目开发（> 1000 行代码）
- 多人协作项目
- 需要长期维护的生产系统

## 不适用场景

- 周末快速原型验证
- 一次性数据处理脚本
- 极小型个人工具（< 1000 行）

---

## 四阶段工作流

| 阶段       | 主导者    | 核心产出      | 关键动作                  |
| ---------- | --------- | ------------- | ------------------------- |
| Specify    | **人**    | spec.md       | 定义问题、边界、成功标准  |
| Plan       | 人 + AI   | plan.md       | 架构选型、模块划分、接口  |
| Implement  | **AI**    | 代码 + 测试   | 按 plan 逐任务实现        |
| Validate   | 人 + AI   | 测试报告      | 自动化测试 + 人工 Review  |

**核心原则：人定义 WHAT，AI 实现 HOW。** 这不是僵硬的瀑布流程，Specify 和 Plan 之间、Implement 和 Validate 之间是持续循环的。

---

## 三文件体系

### 1. spec.md —— 需求规格（唯一真实来源）

必须包含六个要素：

| 要素                  | 作用                 | 示例                                    |
| --------------------- | -------------------- | --------------------------------------- |
| Problem Statement     | 定义「为什么做」     | "当前系统不支持细粒度权限控制"          |
| Success Metrics       | 定义「做到什么程度」 | "P95 < 50ms，覆盖 20+ 权限类型"         |
| User Stories          | 定义「谁在什么场景」 | "作为管理员，我可以创建自定义角色"      |
| Acceptance Criteria   | 定义「怎么验证」     | "单用户多角色，权限取并集"              |
| Non-Goals             | 定义「什么不做」     | "本期不做跨组织权限委托"                |
| Constraints           | 定义「技术约束」     | "必须兼容现有 OAuth2.0 流程"            |

### 2. plan.md —— 架构方案

基于 spec.md 生成，由 AI 起草、人审核修改。包含：

- Architecture Decision（架构决策）
- Module Breakdown（模块划分）
- Interface Contracts（接口契约）
- Risk Assessment（风险评估）

### 3. tasks.md —— 任务清单

将 plan 拆解为可独立验证的原子任务：

- 每个任务对应一个可交付的增量
- 每个任务有明确的验证标准
- 支持 AI 并行执行

## constitution.md —— 项目宪法

定义不可违背的约束，所有 Spec 必须遵守：

```markdown
### 示例条款
- 所有 API 遵循 RESTful 规范，版本化路径 (/api/v1/...)
- 所有用户输入必须经过校验和清洗
- 数据库查询必须使用参数化查询，禁止字符串拼接
- 单元测试覆盖率不低于 80%
- 配置项通过环境变量注入，禁止硬编码
```

**价值：** 将团队技术决策固化为 AI 的「潜意识」，避免每个 Spec 重复声明基本约束。

## 好 Spec vs 坏 Spec

### 坏 Spec ❌

```
"系统需要一个快速的搜索功能。界面要美观易用。"
```

**致命问题：** 模糊、"快速"是多快？遗漏边界（搜索什么？）、缺乏理由、混入 HOW。

### 好 Spec ✅

```markdown
## Problem Statement
用户反馈在 10,000+ 文档中查找目标文档平均需要 3 分钟。
目标是将查找时间缩短到 10 秒以内。

## Success Metrics
- 搜索 API P95 < 200ms
- Top-5 相关性准确率 > 85%（基于人工标注测试集）
- 支持中英文混合查询

## Non-Goals
- 不实现语义搜索（本期仅关键词匹配 + 分词）
- 不支持搜索附件内容（仅标题和正文）
```

**差异本质：好 Spec 是可测试的，坏 Spec 是可解释的。**

---

## 粒度检验标准

> **「用不同技术栈实现这个 Spec，Spec 是否仍然有效？」**

- ✅ 有效 → 是正确的粒度（只定义 WHAT）
- ❌ 失效 → 混入了 HOW，需要修改

示例：
- ❌ "使用 Redis 的 ZSET 结构存储排行榜" → 换成 PostgreSQL 就失效，属于 HOW
- ✅ "排行榜需支持实时更新，延迟 < 1s，支持 Top-100 查询" → 无论底层用什么，Spec 都成立

## Spec 迭代规律

**Spec 编写通常需要 3-5 次迭代才合格。**

1. 写第一版 Spec
2. 让 AI 基于 Spec 生成 plan
3. 回头审视 Spec，暴露盲点
4. 修改 Spec → 重新 Plan
5. 重复 3-5 轮直到合格

**目的：** 把「开发到一半发现需求有问题」的代价前移到成本最低的阶段。改一行 Spec 的成本远低于改一百行代码。

## SDD vs Vibe Coding

| 维度       | Vibe Coding              | SDD                        |
| ---------- | ------------------------ | -------------------------- |
| 核心假设   | AI 能理解你的意图        | AI 需要明确规格才能正确    |
| 启动速度   | 极快                     | 较慢（需先写 Spec）        |
| 可维护性   | 差（无文档、无约束）     | 好（Spec 即文档）          |
| 可协作性   | 差（只有作者知道 vibes） | 好（Spec 是共享语言）      |
| 安全性     | 差（45% 安全漏洞）       | 较好（Spec 约束安全边界）  |
| 适用规模   | < 1000 行                | 中大型项目                 |
| 天花板     | **三个月墙**             | 取决于 Spec 体系质量       |

### 「三个月墙」警告

| 阶段   | 时间        | 状态 | 典型表现                       |
| ------ | ----------- | ---- | ------------------------------ |
| 兴奋期 | 1-3 个月     | 高产出 | "AI 太神了！一天搞定一个功能！" |
| 平台期 | 4-9 个月     | 停滞  | "为什么新功能总是破坏旧功能？"  |
| 衰退期 | 10-15 个月   | 崩溃  | "这坨代码没人能维护了，不如重写" |

---

## 混合策略（实用建议）

1. **探索阶段用 Vibe Coding** —— 快速试错，验证可行性
2. **一旦决定要做，立刻补 Spec** —— 把探索发现固化为规格
3. **进入迭代后严格执行 SDD** —— Spec 驱动，AI 执行

## 一句话总结

> **SDD 是对思考的版本控制。** 当代码可以被 AI 秒级重写时，真正有价值的不是代码本身，而是代码背后的决策。
