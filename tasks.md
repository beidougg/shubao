## 2. Phase 1：基础设施搭建（P0）

**目标**：跑通最小闭环 — 微信发链接 → 抓取 → 摘要 → 存储

### 任务列表

| ID | 任务 | 依赖 | 验收标准 |
|----|------|------|---------|
| T1.1 | 安装配置 OpenClaw | 无 | `openclaw gateway` 正常运行 |
| T1.2 | 接入 Hermes 4（OpenRouter API） | T1.1 | 通过 OpenClaw 成功调用 Hermes 4 模型 |
| T1.3 | 配置微信渠道 | T1.1 | 微信扫描二维码登录成功，收到测试消息 |
| T1.4 | 建立知识库目录结构 | 无 | `~/knowledge-base/` 下 6 大类目录创建完毕 |
| T1.5 | 配置 Memory Markdown 存储路径 | T1.4 | OpenClaw agent workspace 指向知识库目录 |
| T1.6 | 端到端走通：微信发链接 → 抓取 → 摘要 → 存储 | T1.2, T1.3, T1.5 | 微信发 URL，知识库自动生成带 Frontmatter 的 Markdown 文件 |

## 3. Phase 2：核心知识流（P0/P1）

**目标**：完整的「投喂 → 分析 → 归类 → 检索」闭环

### 任务列表

| ID | 任务 | 依赖 | 验收标准 |
|----|------|------|---------|
| T2.1 | 开发 URL 抓取 + 内容清洗 Skill | T1.6 | 任意网页 URL 能提取出干净的正文文本 |
| T2.2 | 开发 AI 分析 Prompt 模板（摘要/分类/标签） | T1.2 | 输入一段文本，输出结构化 JSON（含摘要、分类路径、5-10 个标签） |
| T2.3 | 实现完整知识录入流程（输入→分析→Markdown→入库） | T2.1, T2.2 | 微信投喂任何内容后，知识库生成格式完整的 Markdown 笔记 |
| T2.4 | 建立分类体系配置文件 `categories.yaml` | T1.4 | 文件符合 4.3 节定义的 YAML schema |
| T2.5 | 实现基本检索（关键字/日期/分类查找） | T2.3 | 在微信中发送查询，返回匹配的笔记列表 |
| T2.6 | 配置 Obsidian Vault + OpenClaw 插件 | T2.3 | Obsidian 打开知识库，支持双向同步 |
| T2.7 | 安装推荐 Obsidian 插件（Dataview/Templater/Calendar） | T2.6 | 各插件可正常使用 |

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| OpenRouter API 不稳定 | 分析链路中断 | Phase 1 期间观察 API 稳定性；后期可切换本地 Ollama |
| 微信灰度未覆盖 | 无法通过微信交互 | 备选 Telegram 渠道，或通过 REST API 测试 |
| 知识条目超 100 后检索变慢 | 用户体验下降 | Phase 3 及时引入 ChromaDB |
| OneNote API 权限复杂 | 导入功能延期 | 不影响核心流程，Phase 4 再处理 |
