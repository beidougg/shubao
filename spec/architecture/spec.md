# Shubao（书宝）— 需求规格说明书

## 1. 业务目标

构建一套个人 AI 知识管理系统，核心能力包括：

- **多渠道输入**：支持文字、链接、图片、音视频、文件等多形态资料投喂
- **存量导入**：支持从 OneNote 等现有笔记系统批量加载历史知识
- **智能分析**：对输入内容自动进行摘要、分类、标签提取、实体识别
- **编译式存储**：采用 LLM Wiki 三层架构（raw/ 不可变来源 + wiki/ LLM 编译维护 + schema/ 行为规范），知识按概念组织而非按来源平铺
- **增量编译**：新来源不是创建孤立文件，而是更新已有概念页、实体页、交叉引用
- **问答检索**：支持自然语言查询，基于已编译的 wiki 而非每次从原始文档重新检索
- **定期归纳 + 健康检查**：按日/周/月自动回顾；Lint 操作检测矛盾、过时内容、孤页、知识盲区
- **可编程扩展**：通过 Skills/插件机制持续扩充系统工具能力

## 3. 输入输出定义

### 3.1 输入渠道与格式

| 渠道 | 数据类型 | 格式 |
|------|---------|------|
| 微信 | 文字、链接、图片、音频、视频、文件 | 微信原生消息类型 |
| REST API | JSON | `{ "type": "url\|text\|file", "content": "..." }` |
| OneNote 导入 | HTML | MS Graph API 返回的页面内容 |

### 3.2 输出形式（更新：三层架构）

| 输出 | 目标位置 | 格式 |
|------|---------|------|
| 原始资料 | `raw/` 目录 | Markdown（网页/转写/OCR 后的纯文本） |
| 来源摘要页 | `wiki/source/` | YAML Frontmatter + Markdown 摘要 |
| 概念页 | `wiki/concept/` | YAML Frontmatter + 综合多来源的结构化知识 |
| 实体页 | `wiki/entity/` | YAML Frontmatter + 实体信息 |
| 页面目录 | `wiki/index.md` | Obsidian [[双链]] 列表 |
| 操作日志 | `wiki/log.md` | 追加式时间线 |
| 对比分析页 | `wiki/compare/` | 对比表格 + 分析 |
| 综合页 | `wiki/synthesis/` | 跨主题综述 |
| 问答回复 | 微信 / `wiki/qa/` | 文本 + 引用来源 |
| 每日/周/月报 | 微信 / `wiki/daily/` | Markdown 格式文本 |
| Lint 报告 | `wiki/meta/` | 矛盾/过时/孤页清单 |
| 关联图谱 | Obsidian Graph View | 通过 [[双链]] 自动生成 |

### 3.3 知识条目输出格式

每篇知识笔记为带 YAML Frontmatter 的 Markdown 文件：

```yaml
---
id: "2026-05-12-001"
title: "Transformer 架构详解"
source: "https://arxiv.org/abs/1706.03762"
source_type: "paper"
created_at: "2026-05-12T10:30:00+08:00"
updated_at: "2026-05-12T10:30:00+08:00"
category: "AI/深度学习"
tags: ["transformer", "attention", "nlp", "论文精读"]
entities: ["Google", "Vaswani", "Self-Attention"]
summary: "Transformer 是一种基于自注意力机制..."
status: "active"
---
```

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 目标 |
|------|------|
| 知识录入耗时 | 从微信投喂到归档 < 30 秒 |
| 检索响应时间 | 精确检索 < 2 秒，语义检索 < 10 秒 |
| 系统可用性 | 7×24 持续运行 |

### 4.2 质量要求

| 指标 | 目标 |
|------|------|
| 检索命中率 | 前 3 条结果包含目标内容 > 90% |
| 分类准确率 | 自动分类与用户预期一致 > 85% |
| 扩展效率 | 新增一个 Skill 的开发时间 < 2 小时 |

### 4.3 安全约束

- 敏感内容不入知识库的过滤机制
- API Key 通过环境变量管理，不写入配置文件
- 微信渠道通过配对码授权，防止陌生人使用
- 知识库文件权限控制（仅本地可读）
- OneNote API Token 仅请求只读权限（Notes.Read）

### 4.4 存储约束（更新：三层架构）

- **raw/** — 不可变原始来源，LLM 只读，人类可手动管理
- **wiki/** — LLM 编译维护的知识网络（概念页、实体页、来源摘要、对比、综合）
- **schema/** — 行为规范（shubao.md + strategy/ 策略文件 + prompts/ + templates/）
- 知识库目录：`~/knowledge-base/`
- 版本控制：Git 管理整个 knowledge-base 仓库
- wiki/ 本地搜索：qmd（BM25 + 向量混合搜索，后期引入）
- 语义索引（后期可选）：ChromaDB / Qdrant

## 5. 约束与假设

### 5.1 技术约束

- Agent 框架：OpenClaw（不可替换）
- AI 推理模型：Hermes 4，仅通过 OpenRouter API 接入（已确认）
- 知识存储：Memory Markdown（不可替换）
- 主要交互渠道：微信（已确认）
- 知识阅读器：Obsidian

### 5.2 运行环境

- 操作系统：macOS
- OpenClaw >= v2026.3.22
- 微信 App iOS >= 8.0.70 / Android >= 8.0.69

### 5.3 待确认事项

- OneNote 存量笔记数量及迁移策略
- Obsidian 使用经验及配置时机
- 孩子教育分类子类是否需要补充
