## 10. 关键设计变更说明

### 旧架构 → 新架构

| 维度 | 旧设计 | 新设计（LLM Wiki） |
|------|--------|-------------------|
| 存储层 | 1 层（知识笔记按分类目录） | 3 层（raw/ 不可变 + wiki/ LLM 维护 + schema/ 规范） |
| 知识组织 | 按来源 1:1（每篇来源一个文件） | 按概念多:多（多个来源综合到一个概念页） |
| 新来源处理 | 创建 1 个新文件 | 更新 5-15 个已有文件 |
| 检索方式 | 文件名 + ChromaDB 向量搜索 | index.md 导航 + qmd 全文搜索 |
| 维护操作 | Ingest + Query | Ingest + Query + Lint + 答案归档 |
| ChromaDB 定位 | Phase 3 必须引入 | 可选，仅在 wiki 规模很大时引入 |
| 行为规范 | 无 | schema/ 三层（shubao.md / strategy/ / prompts/） |

### 为什么 ChromaDB 不再紧急

LLM Wiki 模式下，知识已经按概念编译好了。当用户问"自注意力机制"，LLM 直接加载 `wiki/concept/自注意力机制.md`，不需要向量搜索。**index.md 就是轻量的路由层。** 只有以下情况才需要 ChromaDB：
- wiki/ 页数超过 200，index.md 太长无法一次加载
- 需要纯语义模糊搜索（"跟 transformer 里那个 QKV 有关的"）
- 跨概念发现隐藏关联

## 11. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| OpenRouter API 不稳定 | Ingest/Query 链路中断 | Phase 1 期间观察稳定性；后期可切换本地 Ollama |
| 微信灰度未覆盖 | 无法通过微信交互 | 备选通过 REST API / CLI 测试 |
| 增量编译质量差 | 概念页混乱、交叉引用断裂 | schema/ 规范 + strategy/ 策略文件约束；Lint 定期修复 |
| 编译信息丢失 | LLM 过度压缩导致关键事实丢失 | raw/ 不可变的原始来源始终可回溯；WiCER 迭代编译（后期） |
| wiki 规模膨胀 | index.md 超长，导航效率下降 | 引入 qmd 搜索作为补充（Phase 3） |
| OneNote API 权限复杂 | 导入功能延期 | 不影响核心流程，Phase 4 再处理 |
