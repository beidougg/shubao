# Shubao（书宝）— 系统设计文档（LLM Wiki 优化版）

## 1. 技术选型

| 组件 | 选型 | 选型理由 |
|------|------|---------|
| Agent 框架 | OpenClaw | 多渠道接入、Memory 持久化、Cron 定时、Skills 扩展、原生微信支持 |
| AI 推理模型 | Hermes 4 (OpenRouter API) | 强推理能力、混合推理模式、原生 Tool Calling、低拒绝率 |
| 知识本体存储 | Memory Markdown | 人可读、Git 版本控制、天然兼容 Obsidian、零门槛 |
| 知识阅读器 | Obsidian | 本地优先、双向链接、图谱可视化、1000+ 插件生态 |
| Wiki 本地搜索 | qmd | 混合 BM25/向量搜索 + LLM 重排序，全设备本地运行 |
| 语义索引（后期） | ChromaDB / Qdrant | 大规模语义相似度搜索、知识关联发现 |
| Embedding 模型（后期） | BGE / text2vec | 中文友好、本地部署 |

### 2.2 三层存储详解

#### 第一层：raw/ — 不可变原始来源

```
~/knowledge-base/raw/
├── articles/          # 网页文章全文 (Markdown)
│   └── 2026-05-12-transformer-详解.md
├── papers/            # 论文
├── books/             # 书籍章节
├── podcasts/          # 播客转写文本
├── videos/            # 视频转录 + 关键帧描述
├── images/            # 图片 OCR + Vision 描述
├── notes/             # 用户手写笔记
├── onenote/           # OneNote 导入原始内容
└── assets/            # 图片/附件本地存储
```

**核心规则：raw/ 是只读的**，LLM 从这里读取但永远不修改。这是知识的事实来源（source of truth）。

#### 第二层：wiki/ — LLM 编译维护的知识网络

```
~/knowledge-base/wiki/
├── index.md                    # 页面目录（每页一行摘要，按分类组织）
├── log.md                      # 追加式操作日志
│
├── concept/                    # 概念页 — 综合多来源的知识
│   ├── transformer 架构.md
│   ├── 自注意力机制.md
│   ├── RAG 技术.md
│   └── ...
│
├── entity/                     # 实体页 — 人物/公司/技术名词
│   ├── Andrej Karpathy.md
│   ├── OpenAI.md
│   └── ...
│
├── source/                     # 来源摘要页 — 每篇原始资料的摘要
│   ├── 2026-05-12-transformer-详解.md
│   └── ...
│
├── compare/                    # 对比分析页
│   ├── Transformer vs RNN.md
│   └── ...
│
├── synthesis/                  # 综合页 — 跨主题的综述
│   ├── LLM 技术全景.md
│   └── ...
│
├── daily/                      # 每日/每周/每月报告
│   ├── 2026-W20-周报.md
│   └── ...
│
├── qa/                         # 好问题+回答归档
│   └── ...
│
└── meta/                       # 元信息
    ├── tag-index.md            # 标签索引
    ├── orphan-pages.md         # 孤页清单
    └── stale-pages.md          # 过时页面清单
```

**核心规则：wiki/ 由 LLM 全权写入和维护。** 人类阅读，LLM 书写。每个新来源不会只产生一个文件，而是通过 Ingest 操作更新 5-15 个已有页面。

#### 第三层：schema/ — 行为规范

```
~/knowledge-base/schema/
├── shubao.md                   # 主规范文件（类似 CLAUDE.md / AGENTS.md）
│   ├── 命名规范
│   ├── Frontmatter 字段要求
│   ├── 页面模板定义
│   ├── 双链语法规范
│   └── 操作流程详细说明
│
├── strategy/                   # 各领域策略文件（解决平铺问题）
│   ├── 01-技术.md              # 哪些概念高价值、哪些是常规
│   ├── 02-产品.md
│   ├── 03-商业.md
│   ├── 04-个人成长.md
│   ├── 05-会议与音视频.md
│   └── 06-孩子教育.md
│
├── prompts/                    # Prompt 模板
│   ├── ingest.md               # Ingest 操作 Prompt
│   ├── query.md                # Query 操作 Prompt
│   └── lint.md                 # Lint 操作 Prompt
│
└── templates/                  # 页面模板
    ├── concept-page.md         # 概念页模板
    ├── entity-page.md          # 实体页模板
    └── source-summary.md       # 来源摘要页模板
```

**核心规则：schema/ 是最重要的层。** 它决定 LLM 是有纪律的 wiki 维护者还是随意的聊天机器人。你和 LLM 共同演化这套规范。

### 2.3 检索架构：index.md 导航 + 全文搜索

```
                    用户查询
                       │
                       ▼
              ┌──────────────────┐
              │  LLM 读 index.md  │  ← 轻量路由层，无需向量数据库
              │  定位相关页面      │
              └──────┬───────────┘
                     │
         ┌───────────┴───────────┐
         ▼                       ▼
  ┌──────────────┐        ┌──────────────┐
  │ 直接加载 wiki 页│        │ qmd 全文搜索  │
  │ (精确到概念页)  │        │ (BM25+向量)  │
  └──────┬───────┘        └──────┬───────┘
         │                       │
         ▼                       ▼
  ┌──────────────────────────────────────┐
  │  Hermes 4 综合回答 + 标注引用来源      │
  │  有价值的答案 → 回写到 wiki/qa/         │
  └──────────────────────────────────────┘
```

> **设计原则**：中等规模（<100 来源，<500 wiki 页）时，index.md + 文件系统导航完全够用。引入 qmd 后可覆盖更大规模。ChromaDB 仅在需要纯语义模糊搜索时引入。

### 3.2 概念页 (concept/) — 核心知识载体

```yaml
---
title: "自注意力机制"

```

**正文结构：**

```markdown
# 自注意力机制

## 一句话定义
自注意力机制允许输入序列中的每个位置...
（来源: [[2026-05-12-transformer-详解]]、[[2026-04-20-attention-is-all-you-need]]）

## 核心原理
### Q、K、V 的计算
...

### 缩放点积注意力
...

## 与其他概念的关系
- 是 [[concept/Transformer 架构]] 的核心组件
- 与 [[concept/交叉注意力]] 的区别在于 Q、K、V 来源
- 被 [[concept/FlashAttention]] 优化了计算效率
- ⚠️ 与 [[concept/图注意力]] 容易混淆（见 [[compare/自注意力 vs 图注意力]]）

## 已知争议/未解决问题
- 长序列下的 O(n²) 复杂度问题
- 是否真正理解了位置关系（见 [[concept/位置编码]]）

## 相关来源
- [[source/2026-05-12-transformer-详解]] — 原始论文
- [[source/2026-04-20-attention-is-all-you-need]] — 经典解读
```

**概念页的价值：**
- **综合而非平铺** — 多篇来源的知识合并到一个概念下，不是 1:1 映射
- **双向链接** — 与相关概念、实体、来源全部互联
- **标注不确定性** — 明确标记争议和推测内容
- **增量更新** — 新来源不会创建新页面，而是更新已有概念页

## 4. 三大核心操作

### 4.1 Ingest（编译输入）

```
用户投喂新内容 → 预处理(抓取/转写/OCR) → 保存到 raw/
    │
    ▼
┌──────────────────────────────────────────────┐
│  读取 schema/shubao.md + schema/strategy/*.md │ ← 获取行为规范和领域策略
└──────────────────────┬───────────────────────┘
                       ▼
┌──────────────────────────────────────────────┐
│  Hermes 4 读取 raw/ 新文件 + wiki/index.md     │
│                                              │
│  1. 提取关键信息（概念、实体、主张、数据）        │
│  2. 对比已有 wiki，发现：                       │
│     - 哪些概念页需要更新                        │
│     - 哪些是新概念（需新建页面）                  │
│     - 是否存在矛盾（标注而非删除）               │
│  3. 根据 strategy/ 文件判断优先级：             │
│     - 高价值概念 → 详细展开                     │
│     - 常规概念 → 简要记录                       │
│  4. 执行更新（一次可能触碰 5-15 个文件）         │
│     - 创建/更新 concept/ 概念页                │
│     - 创建/更新 entity/ 实体页                 │
│     - 创建 source/ 来源摘要页                  │
│     - 更新 index.md                           │
│     - 追加 log.md                             │
│     - 必要时创建 compare/ 对比页               │
└──────────────────────────────────────────────┘
```

**与旧设计的核心区别：** 不再是一次输入 → 一个文件。而是一次输入 → 更新整个知识网络。LLM 是"编译器"，把原始资料编译进已有知识结构。

### 4.2 Query（检索问答）

```
用户提问
    │
    ▼
┌──────────────────────────────┐
│  Hermes 4 读取 wiki/index.md  │  ← 快速定位相关页面
│  确定需要加载哪些 wiki 页面     │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│  加载相关 wiki 页面（完整文件）  │  ← 不是 chunk，是完整的已编译知识
│  必要时回退到 raw/ 查原文      │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│  Hermes 4 综合回答            │
│  - 综合多个概念页的信息        │
│  - 标注引用来源（wiki 页面）    │
│  - 标注不确定性（如果有）       │
└──────────────┬───────────────┘
               ▼
┌──────────────────────────────┐
│  判断：这个回答有价值吗？       │
│  是 → 回写 wiki/qa/ 或更新    │
│       已有概念页              │
│  否 → 仅返回给用户            │
└──────────────────────────────┘
```

**关键优化：**
- 回答在已编译的 wiki 上做，而不是每次都从 raw/ 重新检索
- 有价值的回答自动归档到 wiki，不让知识散落在聊天记录里
- LLM Wiki 的检索精度高于 RAG，因为读的是完整的概念页而非碎片化的 chunk

### 4.3 Lint（健康检查）

```
定时触发 或 手动触发
    │
    ▼
┌──────────────────────────────────────────────┐
│  Hermes 4 系统性检查 wiki/                     │
│                                              │
│  1. 矛盾检测：两页对同一事实描述不一致          │
│  2. 过时检测：新来源已推翻旧主张，但旧页未更新    │
│  3. 孤页检测：没有入链的页面                   │
│  4. 缺失交叉引用：提到但未链接的概念            │
│  5. 知识盲区：多次出现但无独立概念页的关键词     │
│  6. 断裂链接：[[双链]] 指向不存在的页面         │
│                                              │
│  输出 → 写入 wiki/meta/stale-pages.md          │
│         生成修复建议，等待用户确认执行           │
└──────────────────────────────────────────────┘
```

**Lint 的价值：** 这是 LLM Wiki 相比传统 wiki 的核心优势。人类维护 wiki 会放弃，因为交叉引用和一致性的维护负担太重。LLM 不知疲倦，可以在一次操作中检查上百个文件。

## 5. 核心流程（更新）

### 5.1 知识录入流程（Ingest）

```
用户投喂内容（微信）
    │
    ▼
┌──────────────┐
│  类型识别     │
└──────┬───────┘
       ▼
┌──────────────┐
│  预处理       │  抓取/转写/OCR → 保存到 raw/
└──────┬───────┘
       ▼
┌──────────────┐
│  Hermes 4     │
│  Ingest 模式   │
│              │
│  读取 schema/ │  ← 遵循规范
│  读取 index.md│  ← 了解现有 wiki 结构
│  读取 strategy│  ← 判断价值优先级
│  读取 raw/    │  ← 消化新来源
│              │
│  提取关键信息  │
│  对比已有 wiki │  ← 找出需要更新的页面
│  增量编译     │  ← 更新 5-15 个文件
│              │
│  输出:       │
│  - 新建/更新 concept/ 页
│  - 新建/更新 entity/ 页
│  - 新建 source/ 页
│  - 更新 index.md
│  - 追加 log.md
└──────┬───────┘
       ▼
┌──────────────┐
```

### 6.2 模块职责

| 模块 | 职责 | LLM Wiki 对应层 |
|------|------|:-------------:|
| Preprocessor | 抓取/转写/OCR，产出写入 raw/ | 预处理 |

## 7. 存储演进策略（更新）

| 阶段 | 存储方案 | 检索方式 | 触发条件 |
|------|---------|---------|---------|
| Phase 1-2 | raw/ + wiki/ (纯 Markdown) | index.md 导航 | 系统起步 |
| Phase 3 | 引入 qmd 本地搜索 | BM25 + 向量混合搜索 | wiki/ 页数 > 50 |
| Phase 4+ | 引入 ChromaDB（可选） | 纯语义搜索 | 需要模糊语义匹配 |

> **核心原则转变**：旧设计将 ChromaDB 视为"语义搜索的必然选择"。新设计认识到——在 wiki 已经按概念组织好的前提下，**index.md + 全文搜索已覆盖大部分场景**。ChromaDB 仅在内容量极大或需要高度模糊匹配时才引入。

### 8.2 Schema 文件：shubao.md

这是 LLM Wiki 的"行为指南"。示例结构：

```markdown
# Shubao Wiki 规范

## 命名规范
- concept/ 下：中文名，如 `自注意力机制.md`
- entity/ 下：原始名称，如 `Andrej Karpathy.md`
- source/ 下：`YYYY-MM-DD-简短描述.md`

## 页面模板
### 概念页必须包含
- 一句话定义（含引用来源）
- 核心原理（2-5 段）
- 与其他概念的关系（双向链接）
- 已知争议/未解决问题
- 相关来源列表

### 实体页必须包含
- 一句话描述
- 关键贡献/角色
- 关联概念/相关实体（双向链接）

## 操作流程
### Ingest
1. 读取新来源 → 保存 source/ 摘要页
2. 识别涉及的概念和实体
3. 对比已有 wiki 页面，判断是更新还是新建
4. 读取 domain strategy/ 文件判断优先级
5. 执行更新（一次可能触碰 5-15 个文件）
6. 更新 index.md + 追加 log.md
7. 向用户报告变更摘要

### Query
1. 读取 index.md 定位相关页面
2. 加载 1-5 个 wiki 页面到上下文
3. 综合回答并标注引用来源
4. 判断回答价值，决定是否回写 wiki

### Lint
1. 全量扫描 wiki/ 目录
2. 检测矛盾、过时、孤页、断裂链接、知识盲区
3. 输出报告到 wiki/meta/

## 双链语法
- 概念之间: `[[concept/XXX]]`
- 引用实体: `[[entity/XXX]]`
- 引用来源: `[[source/XXX]]`
- 对比分析: `[[compare/XXX]]`
```

### 8.3 领域策略文件（Strategy per Domain）

每个知识领域一个 `schema/strategy/XX.md`，告诉 LLM 哪些概念值得深入展开：

```markdown
# 技术领域策略

## 高价值概念（详细展开，主动建关联）
- Transformer 架构及变体
- RAG / Agent 架构
- 模型训练与微调方法
- 推理优化（量化、KV Cache、FlashAttention）

## 常规概念（简要记录即可）
- 编程语言语法细节
- 工具使用技巧
- 框架 API 文档

## 不归档内容
- 已过时的技术（被明确替代的）
- 纯个人偏好/吐槽类内容
```

> **为什么需要策略文件？** 来自社区反馈：没有策略文件时，LLM 对待所有概念一律平等，导致 wiki 充斥大量低价值页面。策略文件让 LLM 知道"什么值得深入，什么点到为止"。

```bash
npx -y @tencent-weixin/openclaw-weixin-cli install

openclaw channels login --channel openclaw-weixin
```

## 11. 安全设计

- 敏感内容不入知识库的过滤机制
- API Key 通过环境变量管理
- 微信渠道通过配对码授权
- 知识库文件仅本地可读
- OneNote API Token 仅 Notes.Read 权限
- raw/ 不可修改，确保原始来源可追溯
